# /chris-guide

Print a comprehensive reference for the Chris workflow system. Use this to brief yourself
or a subagent on the full system. No arguments needed.

---

## What Chris Is

Chris is a personal workflow manager for one developer. It takes ideas from first thought to
shipped work through a fixed, repeatable pipeline ‚Äî managing git, spawning agents, and
coordinating work across multiple repos and concurrent projects.

Chris supports 6 project types with two pipeline tracks:

**Code projects** (7 stages):
```
/wf-new ‚Üí [/wf-prd-research] ‚Üí /wf-prd ‚Üí [/wf-spec-research] ‚Üí /wf-spec ‚Üí /wf-tasks ‚Üí /wf-build ‚Üí /wf-review ‚Üí /wf-done
```

**Non-code projects** (5 stages):
```
/wf-new ‚Üí /wf-plan ‚Üí /wf-build ‚Üí /wf-review ‚Üí /wf-done
```

Everything is markdown + Claude CLI skills. No compiled app. Not multi-user.

| Type | Pipeline | Description |
|------|----------|-------------|
| `code` | 7-stage | Software implementation with PRD ‚Üí SPEC ‚Üí TASKS |

| `investigation` | 5-stage | Data-driven diagnosis (debugging, root-cause analysis) |
| `writing` | 5-stage | Long-form content drafting |
| `communication` | 5-stage | Message drafting for Slack, email, Notion |

## Workflow Stages

### Code projects

| Stage | Command | What it does | Output |
|-------|---------|-------------|--------|
| `new` | `/wf-new [name]` | Create project registry entry | `status.json` |
| `prd-research` | `/wf-prd-research` | Market/competitive research before PRD (optional) | `PRD-RESEARCH.md` |
| `prd` | `/wf-prd` | Write PRD interactively, section by section | `PRD.md` |
| `spec-research` | `/wf-spec-research` | Technical design research before spec (optional) | `SPEC-RESEARCH.md` |
| `spec` | `/wf-spec` | Generate technical spec from PRD | `SPEC.md` |
| `tasks` | `/wf-tasks` | Break spec into ordered tasks; set up branches | `TASKS.md` |
| `build` | `/wf-build` | Spawn agent on next unchecked task | Running agent |
| `review` | `/wf-review` | Review diff vs spec + tasks; push + draft PR | Review report |
| `done` | `/wf-done` | Release artifacts + git cleanup + archive | `release/` |

### Non-code projects

| Stage | Command | What it does | Output |
|-------|---------|-------------|--------|
| `new` | `/wf-new [name]` | Create project registry entry | `status.json` |
| `plan` | `/wf-plan` | Write plan using type-specific template | `PLAN.md` |
| `build` | `/wf-build` | Route to type-specific build handler | Running agent |
| `review` | `/wf-review` | Type-aware review of deliverables | `REVIEW.md` |
| `done` | `/wf-done` | Release artifacts + archive | `release/` |

### Supporting skills (any stage)

- `/wf-status [all] [--dashboard]` ‚Äî list all projects; `--dashboard` writes Obsidian markdown
- `/wf-research [topic]` ‚Äî run research and save to project's `research/`
- `/wf-investigate [slug]` ‚Äî run next investigation step (investigation projects)
- `/wf-communicate [slug]` ‚Äî draft communication with approval gate (communication projects)

You don't have to go in order. Skip stages that don't apply. Re-run any stage to update it.

### Vault layout (when configured)

When `vault_path` is set in `~/.chris/config.yml`, `~/Code/chris/projects/` is a directory-level symlink to `<vault_path>/Projects/`. All project data lives in the vault and is tracked by its own git repo:

```
<vault_path>/
‚îú‚îÄ‚îÄ .stignore                      ‚Üê excludes Projects/.git from Syncthing
‚îú‚îÄ‚îÄ dashboard.md                   ‚Üê generated by /wf-status --dashboard
‚îî‚îÄ‚îÄ Projects/                      ‚Üê git repo (chris-projects), symlinked from ~/Code/chris/projects
    ‚îî‚îÄ‚îÄ <slug>/                    ‚Üê same structure as above
```

No per-project symlinks are needed ‚Äî the single directory-level symlink handles everything transparently.

## Agent Specs

Agent behaviour specs live in `agents/`. Each defines role, inputs, outputs, tools, and constraints.

| Agent | Purpose | Spawned by |
|-------|---------|------------|
| `research-analyst` | Deep research with source citations | `/wf-build` (research), `/wf-research` |
| `investigator` | Data-driven investigation with evidence chains | `/wf-investigate` |
| `writer` | Long-form drafting by outline section | `/wf-build` (writing), `/wf-write` |
| `communicator` | Message drafting for Slack/email/Notion | `/wf-communicate` |

Agents are spawned by skills, not invoked directly. They write output to project directories and produce handoff JSON files.

## AGENTS.md ‚Äî Repo Brief Format

Every code repo has `AGENTS.md` at its root. Agents load this at session start.

```yaml
---
name: My API
slug: my-api
repo: ~/Code/my-api
stack: [TypeScript, Bun]
stage: build
install_cmd: bun install
default_branch: main
---

## Purpose
What this repo does and why it exists.

## Current Focus
What's being worked on right now (keep this current).

## Conventions
Key patterns, naming, architecture decisions to follow.

## Key Files
- src/index.ts ‚Äî entry point
- src/api/ ‚Äî API layer

## Open Questions
Things that still need decisions.
```

`/wf-new` creates this file. For existing repos: copy from `~/Code/chris/templates/AGENTS.md`.
Stale AGENTS.md is worse than no AGENTS.md ‚Äî keep it updated as the project evolves.

## TASKS.md Format

```markdown
## Phase 1 ‚Äî Foundation

- [ ] TASK-001: Short descriptive title
  **Repos:** repo-name
  **Deps:** none

  Description: what needs to be done.

  **Accepts:** A specific, verifiable condition proving this task is complete.
```

Rules:
- Zero-padded three-digit numbers: TASK-001, TASK-002, etc.
- Ordered so dependencies are satisfied before dependents
- Each task should be completable in a single focused agent session
- Tasks touch as few repos as possible
- Mark complete with `[x]` when done

## Git Operations

Chris manages git for code projects throughout the workflow:

**Branch creation** (`/wf-tasks`):
```bash
git -C ~/Code/<repo> checkout -b chris/<slug>
```

**Worktree setup** (`/wf-build`, when two projects share a repo):
```bash
git -C ~/Code/<repo> worktree add ~/Code/.chris-worktrees/<slug>/<repo>/ chris/<slug>
```

**During build** (agent instructions):
- Commit at logical checkpoints using conventional commits
- Stay on `chris/<slug>` ‚Äî do not switch branches
- Do not push ‚Äî `/wf-review` handles pushing

**On review pass** (`/wf-review`):
```bash
git -C <path> push origin chris/<slug>
gh pr create --head chris/<slug> --base main --draft
```

**Cleanup** (`/wf-done`):
```bash
git -C ~/Code/<repo> worktree remove ~/Code/.chris-worktrees/<slug>/<repo>/ --force
git -C ~/Code/<repo> branch -d chris/<slug>
```

Non-code projects skip all git operations (no branches, worktrees, or PRs).

## Conflict Detection

`/wf-build` checks for conflicts before spawning an agent (code projects only):

1. Find all other `chris/*` branches on the same repo
2. Get files modified by each competing branch since diverging from main
3. Get files modified by the current branch
4. If overlap ‚Üí notify user + add to `status.json` conflicts array (resolved: false)

Chris notifies but does not block. Watch for merge conflicts at PR time.
`/wf-review` surfaces any unresolved conflicts in the review report.

## Project Detection (all wf-* skills)

1. `$ARGUMENTS` provides a slug ‚Üí use it directly
2. cwd is `~/Code/<repo>/` or subdirectory ‚Üí scan all `status.json` files for this repo in `repos` array
   - One match ‚Üí use it
   - Multiple matches ‚Üí ask: "Multiple projects reference this repo: [list]. Which one?"
3. No match ‚Üí list projects eligible for the current stage, ask user to choose

## Vault Backing (optional)

When `vault_path` is set in `~/.chris/config.yml`, `~/Code/chris/projects/` is a symlink to `<vault_path>/Projects/`. All project data lives in the Obsidian vault and is tracked by its own git repo (`chris-projects`).

- **Setup:** Run `scripts/install.sh` and provide your vault path when prompted. The script creates the symlink, initializes the git repo, and adds a `.stignore` to exclude `.git/` from Syncthing.
- **Git:** Pull latest projects with `git -C ~/Code/chris/projects pull`. Push with `git -C ~/Code/chris/projects push`.
- **Dashboard:** `/wf-status --dashboard` writes an Obsidian wiki-link dashboard to `<vault_path>/dashboard.md`.

## Setup

Run `scripts/install.sh` to set up Chris. This replaces the old `/wf-init` skill.

The script:
1. Creates `~/.chris/config.yml` (vault path, AgentOS path)
2. Symlinks all skills to `~/.claude/commands/`
3. Symlinks all agent specs to `~/.claude/agents/`
4. Creates vault directories (if configured)
5. Checks `gh auth status`

## Release Artifacts (generated by /wf-done)

Saved to the project's `release/` directory:

- `PRESS-RELEASE.md` ‚Äî human-readable announcement: headline, what was built, why it matters, key highlights
- `RELEASE-NOTES.md` ‚Äî technical changelog: completed tasks, decisions, deferred items, known issues
- `UPDATE-LOG.md` ‚Äî terse commit log grouped by repo (code projects only)

## Common Gotchas

- **Stale AGENTS.md** ‚Äî update it whenever the project focus changes; agents rely on it
- **Skipping /wf-prd** ‚Äî even a rough PRD catches assumptions before they become bugs
- **Forgetting /wf-review** ‚Äî takes 2 minutes; catches drift from the original spec
- **Running builds in the main checkout when a worktree exists** ‚Äî check `status.json` worktrees first
- **Pushing before review** ‚Äî let `/wf-review` handle pushing; it sets up the PR correctly
- **Using /wf-prd for non-code projects** ‚Äî use `/wf-plan` instead; it loads type-specific templates
- **Sending without approval** ‚Äî communication projects require two explicit approvals before any send
